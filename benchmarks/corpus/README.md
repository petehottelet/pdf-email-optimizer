# Real-world benchmark corpus

This directory documents the categories of real-world PDFs that `pdf-email-optimizer` is validated against. It is the production sibling of `benchmarks/fixtures/`, which contains the small, regenerable, CC0 synthetic fixtures used in CI.

Two intents:

- `public/` holds redistributable PDFs (CC0, public domain, generated, or explicitly licensed for redistribution) along with `notes.md` files describing each fixture. These can be committed to git when they exist.
- `private/` is a documentation placeholder. Maintainers and contributors with PDFs that cannot be shared publicly (client work, internal docs, NDA material) can drop them here locally; the directory is `.gitignore`d. See [`private/README.md`](private/README.md).

The corpus manifest is [`corpus.yaml`](corpus.yaml).

## Each fixture should document

| Field | Required | Notes |
|---|---|---|
| `id` | yes | Snake-case identifier; matches manifest `id` |
| `category` | yes | e.g. `photo_heavy`, `screenshot_heavy`, `design_export`, `scanned`, `text_vector`, `lossless_images`, `forms_annotations`, `transparency`, `acrobat_optimized` |
| `description` | yes | One-line, human-readable |
| `license` | yes | "CC0 / synthetic", "Used with permission", "Public domain", etc. Never commit copyrighted material without permission. |
| `source` | yes | Where it came from (or "generated by benchmarks/make_fixtures.py") |
| `expected_behavior` | yes | What good output looks like: `hit_target_or_warn`, `preserve_vectors`, `protect_screenshots`, `image_recompress`, `structural_cleanup`, etc. |
| `command` | recommended | Exact CLI command used for the benchmark |
| `notes` | recommended | Visual notes, edge cases, known traps |

## Target categories

The validation sprint targets at least 12 categories covering the situations users actually hit when emailing PDFs:

| Category | Why it matters |
|---|---|
| Canva-style marketing export | Common email attachment problem |
| InDesign export | Professional design workflow |
| Illustrator / vector-heavy PDF | Can break naive optimizers |
| Screenshot-heavy report | Common business deck/report |
| Scanned document | Office workflow |
| PDF with forms | Must not silently break fields |
| Annotated PDF | Comments and highlights matter |
| Transparent graphics | Common rendering edge case |
| Image-heavy brochure | Headline compression use case |
| Mixed text + photos | Tests quality balance |
| Already-optimized PDF | Ensures no harmful recompression |
| Encrypted / unlocked workflow | Documents the limitation |

## Adding a fixture

1. Confirm the file is safe to share. Use the [fixture submission issue template](../../.github/ISSUE_TEMPLATE/fixture-submission.yml) if it came from a contributor.
2. Drop the PDF into `public/<id>/input.pdf` along with a `notes.md` capturing the fields above.
3. Add an entry to [`corpus.yaml`](corpus.yaml).
4. Re-run `python benchmarks/run_benchmarks.py` so `benchmarks/results/latest.{json,md,csv}` reflect it.
